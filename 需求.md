# 🤖 AI Agent 指令：Anna App (Mini Notes) 全栈开发规范

## 1. 角色与目标 (Role & Objective)
你是一个精通 **Anna App 平台架构** 的全栈开发专家。你的任务是实现一个名为 **"Mini Notes with LLM Summary"** 的本地 Anna App。
**核心挑战**：这不是一个传统的 Web App。你必须严格遵守 Anna 的“寄生与托管”平台模型，禁止使用传统的 Web 开发范式（如 `localStorage`、直接 `fetch` LLM API、自建 HTTP 后端）。

## 2. 核心架构与“反常识”约束 (CRITICAL CONSTRAINTS)
⚠️ **AI Agent 必须遵守以下绝对规则，否则代码将被直接拒收：**
1. **禁止本地存储**：前端**绝对禁止**使用 `localStorage`、`IndexedDB` 或 React/Vue State 作为数据源。所有笔记的增删改查**必须**通过 `anna.storage.*` Host API 与 Anna 平台交互。
2. **禁止直连 LLM**：前端和后端**绝对禁止**持有任何 LLM API Key，**禁止**直接 `fetch` OpenAI/Claude API。总结功能**必须**通过 Executa Tool 发起 Reverse RPC (`sampling/createMessage`) 向 Anna Host “借用”大脑。
3. **Stdio 纯净性**：Node.js 后端（Executa Tool）的 `stdout` **只能**输出 JSON-RPC 2.0 格式的字符串。任何 `console.log`、banner、debug 信息**必须**输出到 `stderr`，否则会导致协议解析崩溃。
4. **长驻进程**：Node.js 后端必须是一个持续监听 `stdin` 的守护进程，处理完一次 `invoke` 后**绝对不能** `process.exit()`，必须等待 EOF。

## 3. 前端工程化规范 (UI Bundle)
前端是一个运行在 Anna iframe 中的 SPA（推荐 React/Vue/Svelte + Vite）。

### 3.1 Host API 调用规范
前端必须通过 `AnnaAppRuntime.connect()` 获取 `anna` 对象，并调用以下 Host API：
*   **读取笔记**: `await anna.storage.get({ key: 'notes' })`
    *   *返回*: `{ value: Note[], exists: boolean }`
*   **保存笔记**: `await anna.storage.set({ key: 'notes', value: notesArray })`
*   **删除笔记**: 更新 `notesArray` 后再次调用 `anna.storage.set`。
*   **调用后端 Tool**: `await anna.tools.invoke({ tool_id: '<your_tool_id>', method: 'summarize', args: { notes: notesArray } })`

### 3.2 Manifest 声明 (`manifest.json`)
App 根目录必须包含 `manifest.json`，声明 UI 权限与依赖：
```json
{
  "schema": 2,
  "name": "mini-notes",
  "version": "1.0.0",
  "required_executas": ["<your_tool_id>"],
  "ui": {
    "bundle": { "entry": "dist/index.html" },
    "host_api": {
      "storage": ["get", "set", "delete", "list"],
      "tools": ["required:*"]
    }
  }
}
```

## 4. 后端规范：Node.js Executa Tool
后端是一个独立的 Node.js CLI 程序，通过 **JSON-RPC 2.0 over stdio** 与 Anna Agent 通信。

### 4.1 协议握手与生命周期
Agent 启动进程后，会按顺序发送以下请求，你的程序必须正确响应：
1.  **`initialize` (v2 协商)**:
    *   *Host 发来的 Request*: `{"jsonrpc":"2.0", "id":1, "method":"initialize", "params":{...}}`
    *   *你的 Response*: 必须返回 `protocolVersion: "2.0"` 和 `client_capabilities: { sampling: {} }` 以启用 Reverse RPC。
2.  **`describe` (声明能力)**:
    *   *你的 Response*: 返回 Manifest，**必须**包含 `host_capabilities: ["llm.sample"]`，并声明 `summarize` 工具及其 `parameters` schema。
3.  **`invoke` (执行逻辑)**:
    *   *Host 发来的 Request*: `{"jsonrpc":"2.0", "id":2, "method":"invoke", "params":{"tool":"summarize", "arguments":{"notes":[...]}}}`
    *   *你的动作*: 拦截此请求，向 Host 发起 Reverse RPC (`sampling/createMessage`)，等待 Host 返回 LLM 结果，最后将结果包装成 `{"success": true, "data": {...}}` 返回给 Host。

### 4.2 Reverse RPC (Sampling) 实现细节
在同一个 `stdin` 流中，你既会收到 Host 发来的 `invoke` 请求，也会收到 Host 对你 `sampling` 请求的响应。
*   **必须维护一个 `pendingRequests` Map**：通过 JSON-RPC 的 `id` 字段来区分“Host 发来的新任务”和“Host 返回的 Sampling 结果”。
*   **Sampling 请求格式 (Tool -> Host)**:
    ```json
    {
      "jsonrpc": "2.0",
      "id": "<generate_unique_uuid>",
      "method": "sampling/createMessage",
      "params": {
        "messages": [{"role": "user", "content": {"type": "text", "text": "Summarize these notes: ..."}}],
        "max_tokens": 500
      }
    }
    ```

## 5. 本地测试与 Mock 机制 (验收核心)
由于本地没有真实的 Anna 账号和云端 LLM，必须实现两条隔离的测试链路：

### 链路 A：UI Harness 测试 (`anna-app dev --no-llm`)
*   **场景**：Host 禁用了 LLM。
*   **预期行为**：用户在 UI 点击 "Summarize"，前端正常调用 `anna.tools.invoke`，但由于 Host 禁用 LLM，前端**必须**捕获并展示类似 `[-32603] harness started with --no-llm` 的错误。
*   **验收点**：证明前端没有绕过 Tool 直接调 LLM，且 `anna.tools.invoke` 链路已打通。

### 链路 B：Backend Sampling Mock (`anna-app executa dev --mock-sampling`)
*   **场景**：使用 `fixture.jsonl` 伪造 Host 的 Sampling 响应。
*   **`fixture.jsonl` 示例**:
    ```json
    {"id":"<match_rpc_id>", "result":{"content":{"type":"text","text":"Mocked Summary: Buy milk and fix bugs."}}}
    ```
*   **验收点**：证明 Node.js 后端正确发起了 `sampling/createMessage`，并正确解析了 Host 的 Reverse RPC 响应。

## 6. 二进制打包与 CI/CD (GitHub Actions)
Anna 平台不接受“源码 + Node.js 环境”的发布方式，必须打包为**独立二进制文件**。

### 6.1 打包规范 (使用 `@yao-pkg/pkg`)
*   使用 `pkg` 将 Node.js 脚本打包为三平台可执行文件：
    *   `darwin-arm64` (macOS Apple Silicon)
    *   `darwin-x86_64` (macOS Intel)
    *   `windows-x86_64` (Windows)
*   **Archive 结构**：必须将二进制文件打包进 `.tar.gz` (macOS) 或 `.zip` (Windows)。
*   **Archive 根目录必须包含 `manifest.json`**，声明 `runtime.binary.entrypoint` 指向打包后的二进制文件。

### 6.2 GitHub Actions Workflow
*   触发条件：`workflow_dispatch` 或 `release`。
*   矩阵构建：在 Ubuntu (交叉编译 Windows) 和 macOS Runner 上分别构建。
*   Smoke Test：在 CI 中通过 `echo '{"jsonrpc":"2.0","method":"describe","id":1}' | ./binary` 验证二进制文件是否能正常输出 JSON-RPC 响应。
*   产物：上传三个平台的 Archive 到 GitHub Release Assets。
